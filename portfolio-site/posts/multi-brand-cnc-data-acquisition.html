Multi-brand CNC data acquisition: Mitsubishi, NUM, Heidenhain, Siemens, Fanuc and beyond

A field-oriented note on CNC data acquisition where each controller family requires a different access path, permission model, SDK behavior and telemetry mapping strategy.

Context

Multi-brand CNC environments are not uniform. One machine may expose a vendor DLL, another may require protocol-level access, another may be better handled through MTConnect, and another may need a gateway or database-side reader. The architecture must support mixed access methods without forcing every controller into the same implementation path.

Access strategy

  • Mitsubishi: native API / DLL exploration, exported function inspection and vendor runtime dependencies.
  • Heidenhain: LSV2 access, controller permissions, machine variables and protocol constraints.
  • NUM: adapter or protocol-specific acquisition depending on the target machine setup.
  • Siemens / Fanuc: gateway, SDK, MTConnect or shop-floor integration path depending on the available interface.

Telemetry model

The collector should not expose raw vendor values directly to the dashboard. It should convert them into consistent machine states, alarm objects, counters, program context and runtime signals.

Vendor signal → Normalized state → Transport payload → MES / Dashboard

Diagnostics

Reverse engineering and diagnostics tools are part of the workflow: DLL export checks, dependency analysis, network tracing, protocol observation and controlled test calls help define what can be read safely and repeatedly.
